# 文書・コメントの記述ルール（詳細）

> 本体（~/.claude/CLAUDE.md）の「文書・コメント」節が要点だけ持ち、詳細と例はここに置く。
> 文書・コメント・README・設計メモを書く前にこれを読む。

## 恒常的な内容に限定する

README・docs・設計メモ・コード/設定コメントは、**いつ読んでも正しい恒常的な内容**だけを書く。次の transient（時間とともに陳腐化する）情報は書かない:

- 移行中の「目標構成」「移行中」「いま X をやっている」等の進行状態
- タスク/PR ロードマップ、PR 番号・チケットID を進捗トラッカーとして書くこと、`✅完了 / 🔄レビュー中 / ⏸保留` 等のステータス
- 「〜まで暫定」「移行後はこうなる」のような時点依存の記述

これらは課題管理（Linear 等）/ PR 説明 / タスクファイルで追跡し、**コミットする文書・コメントには持ち込まない**。

- **Why**：文書・コメントは特定タスクより長く生きる。transient な内容は完了と同時に陳腐化し、後から読む人を誤導する。
- **例外**：設計判断の理由（「Y だから X を選んだ」）や、理由付きの恒久的な既知制約は恒常的なので可。タスクのステータスメモは不可。
- **判断基準**：「これは半年後でも、今の作業に関係なく正しいか？」を自問する。現在のタスク/PR/移行状態を指すなら Linear/PR へ移す。

## 重複させない（単一ソース化）

同じ内容を複数の文書に書かない。各事実は1箇所（単一の正本）に置き、他からはリンクで参照する。例: 概要・構成図・一覧は README に置き、詳細ドキュメントは README に無いことだけを書いて README を参照する。重複は片方の更新漏れで矛盾を生む。

## 実装の詳細を書かない

文書・コメントには、それを読む人がその場で必要としない実装の内部メカニズムを書かない。次のものは削る:

- 内部動作のメカニズム（例: 「startup probe を同期実行しコンテナが `/health` を返すまで待つ」「`/secrets/.env` にマウント」等）。読み手が知るべきは「何を・どの順で・なぜ」であって「内部でどう動くか」ではない。
- コード・IaC・生成物で定義済みの事実の列挙（生成リソース名・付与ロール一覧・作られるリソースの逐一列挙）。これらは定義元（`*.tf` 等）が単一ソース。文書側で重複させると更新漏れで陳腐化する。

- **何を残すか**：手順（コマンド・順序・確認方法）と、設計判断の理由（「Y だから X」）。理由は概念レベルに圧縮し、内部実装の挙動には踏み込まない。
- **Why**：実装の詳細は実装変更で容易に陳腐化し、定義元と重複して矛盾を生む。文書は「読む人の意思決定に必要な情報」に絞ると、長く正しく保てる。
- **判断基準**：「これは実装が変わっても正しいか？ 定義元（コード/IaC）に既にあるか？」を自問する。実装変更で壊れる or 既に他にあるなら書かない。

## コメントのマーカー記法

- **`TODO` / `FIXME` などのマーカーは必ずコロン付きで書く**（`TODO:` / `FIXME:` / `NOTE:` / `HACK:`）。エディタの構文ハイライトはコロン付きの形を検出するため、`TODO` 単体では色が付かない。
- 補足情報（チケットID・条件など）はコロンの後に続ける。例: `# TODO: (MOV-677) foundation 分離後は不要`。
- **Why**：ハイライトが効くとレビュー時・grep 時に未処理箇所を見落としにくい。
