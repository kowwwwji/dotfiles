## ワークフロー設計

### 1. Planモードを基本とする

- 3ステップ以上 or アーキテクチャに関わるタスクは必ずPlanモードで開始する
- 途中でうまくいかなくなったら、無理に進めずすぐに立ち止まって再計画する
- 構築だけでなく、検証ステップにもPlanモードを使う
- 曖昧さを減らすため、実装前に詳細な仕様を書く

### 2. サブエージェント戦略

- メインのコンテキストウィンドウをクリーンに保つためにサブエージェントを積極的に活用する
- リサーチ・調査・並列分析はサブエージェントに任せる
- 複雑な問題には、サブエージェントを使ってより多くの計算リソースを投入する
- 集中して実行するために、サブエージェント1つにつき1タスクを割り当てる

### 3. 自己改善ループ

- ユーザーから修正を受けたら必ず `tasks/lessons.md` にそのパターンを記録する
- 同じミスを繰り返さないように、自分へのルールを書く
- ミス率が下がるまで、ルールを徹底的に改善し続ける
- セッション開始時に、そのプロジェクトに関連するlessonsをレビューする

### 4. 完了前に必ず検証する

- 動作を証明できるまで、タスクを完了とマークしない
- 必要に応じてmainブランチと自分の変更の差分を確認する
- 「スタッフエンジニアはこれを承認するか？」と自問する
- テストを実行し、ログを確認し、正しく動作することを示す

### 5. エレガントさを追求する（バランスよく）

- 重要な変更をする前に「もっとエレガントな方法はないか？」と一度立ち止まる
- ハック的な修正に感じたら「今知っていることをすべて踏まえて、エレガントな解決策を実装する」
- シンプルで明白な修正にはこのプロセスをスキップする（過剰設計しない）
- 提示する前に自分の作業に自問自答する

### 6. 自律的なバグ修正

- バグレポートを受けたら、手取り足取り教えてもらわずにそのまま修正する
- ログ・エラー・失敗しているテストを見て、自分で解決する
- ユーザーのコンテキスト切り替えをゼロにする
- 言われなくても、失敗しているCIテストを修正しに行く

## タスク管理

1. **まず計画を立てる**：チェック可能な項目として `tasks/todo.md` に計画を書く
2. **計画を確認する**：実装を開始する前に確認する
3. **進捗を記録する**：完了した項目を随時マークしていく
4. **変更を説明する**：各ステップで高レベルのサマリーを提供する
5. **結果をドキュメント化する**：`tasks/todo.md` にレビューセクションを追加する
6. **学びを記録する**：修正を受けた後に `tasks/lessons.md` を更新する

## コア原則

- **シンプル第一**：すべての変更をできる限りシンプルにする。影響するコードを最小限にする。
- **手を抜かない**：根本原因を見つける。一時的な修正は避ける。シニアエンジニアの水準を保つ。
- **影響を最小化する**：変更は必要な箇所のみにとどめる。バグを新たに引き込まない。

## ドキュメント・コメント

- **恒常的な内容に限定する**：README・docs・設計メモ・コード/設定コメントは、**いつ読んでも正しい恒常的な内容**だけを書く。次の transient（時間とともに陳腐化する）情報は書かない:
  - 移行中の「目標構成」「移行中」「いま X をやっている」等の進行状態
  - タスク/PR ロードマップ、PR 番号・チケットID を進捗トラッカーとして書くこと、`✅完了 / 🔄レビュー中 / ⏸保留` 等のステータス
  - 「〜まで暫定」「移行後はこうなる」のような時点依存の記述
- これらは課題管理（Linear 等）/ PR 説明 / タスクファイルで追跡し、**コミットする文書・コメントには持ち込まない**。
- **Why**：文書・コメントは特定タスクより長く生きる。transient な内容は完了と同時に陳腐化し、後から読む人を誤導する。
- **例外**：設計判断の理由（「Y だから X を選んだ」）や、理由付きの恒久的な既知制約は恒常的なので可。タスクのステータスメモは不可。
- **判断基準**：「これは半年後でも、今の作業に関係なく正しいか？」を自問する。現在のタスク/PR/移行状態を指すなら Linear/PR へ移す。
- **重複させない（単一ソース化）**：同じ内容を複数の文書に書かない。各事実は1箇所（単一の正本）に置き、他からはリンクで参照する。例: 概要・構成図・一覧は README に置き、詳細ドキュメントは README に無いことだけを書いて README を参照する。重複は片方の更新漏れで矛盾を生む。
- **実装の詳細を書かない**：文書・コメントには、それを読む人がその場で必要としない実装の内部メカニズムを書かない。次のものは削る:
  - 内部動作のメカニズム（例: 「startup probe を同期実行しコンテナが `/health` を返すまで待つ」「`/secrets/.env` にマウント」等）。読み手が知るべきは「何を・どの順で・なぜ」であって「内部でどう動くか」ではない。
  - コード・IaC・生成物で定義済みの事実の列挙（生成リソース名・付与ロール一覧・作られるリソースの逐一列挙）。これらは定義元（`*.tf` 等）が単一ソース。文書側で重複させると更新漏れで陳腐化する。
- **何を残すか**：手順（コマンド・順序・確認方法）と、設計判断の理由（「Y だから X」）。理由は概念レベルに圧縮し、内部実装の挙動には踏み込まない。
- **Why**：実装の詳細は実装変更で容易に陳腐化し、定義元と重複して矛盾を生む。文書は「読む人の意思決定に必要な情報」に絞ると、長く正しく保てる。
- **判断基準**：「これは実装が変わっても正しいか？ 定義元（コード/IaC）に既にあるか？」を自問する。実装変更で壊れる or 既に他にあるなら書かない。

### コメントのマーカー記法

- **`TODO` / `FIXME` などのマーカーは必ずコロン付きで書く**（`TODO:` / `FIXME:` / `NOTE:` / `HACK:`）。エディタの構文ハイライトはコロン付きの形を検出するため、`TODO` 単体では色が付かない。
- 補足情報（チケットID・条件など）はコロンの後に続ける。例: `# TODO: (MOV-677) foundation 分離後は不要`。
- **Why**：ハイライトが効くとレビュー時・grep 時に未処理箇所を見落としにくい。

### 文書群の役割分担（README = 索引、詳細 doc = 主題のみ）

複数の文書がある場合、各文書の役割を固定して重複を防ぐ。

- **入口は1つ（README 等の索引文書）**：概要・構成図・コンポーネント一覧・環境一覧、および各詳細 doc への**リンク＋1行説明**を置く。読者の入口はここに集約する。
- **詳細 doc は自分の主題だけを書く**：README にある概要・一覧を再掲しない。**自分の守備範囲の再説明もしない**（節見出しが目次を兼ねる）。冒頭に「本書は〜を扱う」という自己紹介列挙を書かない（索引側の1行説明と重複する）。
- **back-link を張らない**：入口（README）→ 各 doc の**一方向リンクのみ**。詳細 doc から README へ戻るリンクは、README が入口である以上ほぼ常に冗長。
- **doc 間の横リンクは最小限**：原則 README 経由で辿らせ、どうしても必要な相互参照だけ張る。
- **「移動先はこちら」ポインタを残さない**：内容を別ファイル（コード/IaC/他 doc）へ単一ソース化した後に「詳細は X を参照」と書き残すのは、旧構成を知る人にしか価値がない transient 情報。ファイル名・節見出しから自明なら消す。
- **Why**：役割を固定しないと、各 doc が概要や一覧を少しずつ再掲し、更新が索引側だけに入って詳細 doc が陳腐化する。「概要・一覧は索引、詳細は該当 doc、実装値はコード/IaC」と置き場所を決め、他はリンクで参照する。

## tmux

- 「tmuxの〇ペインを見て」と言われたら、現在のtmuxセッションの該当ペインを `tmux capture-pane -t {session}:{window}.{pane} -p` で取得する
- ペイン番号だけ指定された場合（例: 「1ペイン」「2ペイン」）は、同セッションのウィンドウ1の該当ペイン（1.1, 1.2, 1.3...）を参照する
- セッション名が不明な場合は `tmux list-panes -a` で一覧を確認して特定する
